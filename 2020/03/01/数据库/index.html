数据库

Author: lpopopo Date: 三月 1, 2020  20:43:43 Category: 数据库

数据库技术即应用

数据库的４个基本概念

  • 数据：是数据库中存储的基本对象

    定义：

    • 描述事件的符号记录
    • 数字，文字，图形，图像，音频，视频，学生的档案记录，订单情况等等

  • 数据库

    • 数据库是长期存储在计算机内，有组织的，可共享的大量数据的集合
    • 建立数据库为了收集并抽取出一个应用所需要的大量的数据，将其保存，以供进一步加工处理．抽取有用的信息，转换为有价值的知识
    • 基本特征：
      • 数据按一定的数据模型组织，描述和存储
      • 可为各种用户共享，冗余度小，易扩展
      • 数据独立性高

  • 数据库管理系统

    • 位于应用与操作系统之间的一层数据管理软件

    • 是基础软件，是一个大型复杂的软件系统

    • 用途：科学的组织和存储数据，高效的获取和维护数据

    • 主要功能：

      • 数据定义功能:

        • 提供数据定义语言(DDL)

        • 定义数据库中的数据对象

        • 数据组织，存储和管理:

          • 分类组织，存储和管理各种数据
          • 确定数据在存储级别上的结构和存取方式
          • 实现数据之间的关系
          • 提供多种存取方法提高存取效率

      • 数据库操作功能：

        • 提供数据操作语言
        • 实现对数据库的基本操作(增，删，改，查)

      • 数据库的事务管理和运行管理

        • 数据的安全性，完整性，多用户对数据的并发使用
        • 发生故障后的系统恢复数据库

  • 数据库系统：是指在计算机系统中引入数据库系统的系统构成

    • 构成：
      • 数据库
      • 数据库管理系统
      • 应用程序
      • 数据库管理员
    • 特点：
      • 数据结构化
      • 数据共享性高，冗余度低且易扩充
        • 物理独立性：指用户的用户程序与数据库中的数据的物理存储是相互独立的
        • 逻辑独立性：应用程序与数据库的逻辑结构是相互独立的
      • 数据独立性高
      • 数据由数据库管理系统统一管理和控制

数据模型

数据模型三大方面要求：

  • 能比较真实的模拟现实世界
  • 容易为人所理解
  • 便于在计算机上实现

两类数据模型：

  • 概念模型，也称为信息模型：主要是按用户的观点来对数据和信息建模，用于数据库设计
  • 逻辑模型和物理模型
    • 逻辑模型：包括网状模型，层次模型，关系模型，面向对象模型，对象关系数据模型，半结构化数据模型
    • 物理模型是对数据最底层的抽象

数据模型的组成要素

  • 数据结构－－描述系统的静态特性
  • 数据操作－－描述系统的动态特性
  • 完整性约束

层次模型

层次模型是用树形结构来表示各类实体以及实体之间的联系

表示方法：
  • 实体性：用记录类型描述，每一个节点表示一个记录类型
  • 属性：用字段描述，每个记录类型可包含若干个字段
  • 联系：用节点之间的连线表示记录类型之间的一对多的父子联系
定义：

满足:

  • 有且只有一个节点没有双亲节点，称为根节点
  • 根以外的其他节点有且只有一个双亲节点

优点：

  • 数据结构简单清晰
  • 查询效率高，性能优于关系模型，不低于网状模型
  • 提供了良好的完整性支持
缺点
  • 结点之间的多对多联系不自然
  • 对插入和删除限制多，应用程序编写复杂
  • 查询子女结点必通过双亲
  • 数据库命令趋于程序化

网状模型

网状模型采用网状结构表示

#####　表示方法(同上)

优点：
  • 能够更为直接的描述现实世界，
  • 具有良好的性能，存取效率较高
缺点
  • 结构复杂，不利掌握
  • DDL DML语言复杂，不容易使用
  • 记录之间通过存取路径实现的，应用程序必须选择存取的路径

关系模型

相关的术语：
  • 关系：一个关系对应通常说是一张表
  • 元组（记录和行）：表中的一行即为一列
  • 属性（列）：表中的一列即为属性名，每个属性的名称为属性名（列名）
  • 主码：也称为码键．表中的某个属性组，他可以唯一的确定一个元组
  • 域：是一组具有相同数据类型的值的集合，属性的取值范围来自某个域
  • 分量：元组中的一个属性值（一条记录中的一个列值）
  • 关系模式：对关系的描述（表头）

关系必须是规范化的：

最基本的规范－关系的每一个分量必须是一个不可分的数据项

数据操作是集合操作，操作对象和结果都是关系，

存取路径对用户隐蔽，提高了数据的独立性，提高了用户的生产率

优点：
  • 建立在严格的数学的概念的基础之上
  • 概念单一
    • 实体和各类联系都用关系来表示
    • 对数据的检索结果也有关系
  • 存取路径对用户透明
    • 具有更高的数据独立性，更好的安全保密性
    • 简化了程序员的工作和数据库开发建立工作
缺点:
  • 存取路径对用户透明，查询效率往往不如格式化数据模型
  • 为提高性能，必须对用户的查询请求进行优化，增加了开发数据库管理的难度

数据库系统的结构

数据库系统的三级模式结构(从内部看)

  • 模式：(也称为逻辑模式)
    • 数据库全体数据的逻辑结构和特征的描述
    • 所有用户的公共数据视图
    • 模式是数据库系统模式结构中心
      • 与数据的物理存储细节和硬件环境无关
      • 与具体的应用程序，开发工具即高级程序语言无关
  • 外模式(也称为子模式或用户模式)　
    • 外模式通常是模式的子集
    • 一个外模式可以为多个应用系统所使用
    • 每个用户只能看见和访问所对应的外模式的数据，简化用户视图
    • 保证数据库安全性的一个有力措施
  • 内模式(也称为存储模式) (应该是数据库的存储引擎)
    • 是数据物理结构和存储方式的描述
    • 数据在数据库内部的表达方式
      • 记录的存储方式：(如，顺序存储，堆存储，按hash方法存储等)
      • 索引的组织方式(B+树，Bitmap , Hash)

数据库系统的组成

硬件平台

  • 足够大的内存
  • 足够打的磁盘或磁盘阵列等外部设备
  • 较高的通道能力，提高数据的传送速率　

关系型数据库

域：

是一组具有相同数据类型的值的集合，就是每一列的取值范围

笛卡尔积：

给定一组域，允许其中的域是相同的，就是每一列的取值范围组成的整张表的集合

关系

是笛卡尔积的子集

码：

码是可以确定一个元组的所有信息的属性名或属性名组

例如在 { a, b, c, d } 中，

假设知道 a 的值就能确定 a, b, c, d 的值，

假设知道 c, d 的值就可以确定 a, b, c, d 的值，

那么 { a } 就是码，{ c, d } 就是码。

并且 { a, b }, { a, c }, { a, b, c }, { a, b, c, d } 等也都是码，因为它们也可以确定一个元组的所有值，即使很多余

候选码：

候选码的真子集中不存在码，候选码可以有多个。

就上面的例子而言，{ a } 是候选码，{ c, d } 是候选码，因为它们的真子集中不存在码。

而诸如 { a, b } 并不是码，因为它的真子集中含有 { a }， 而 { a } 是码

主码：

主码就是主键的意思，主码是任意一个候选码。

还是上面的例子，主码是候选码 { a }， { c, d } 中的其中一个。

既可以是 { a }， 也可以是 { c, d }

基本关系的性质：

  • 列是同质的
  • 不同的列可出自同一域
  • 列的顺序无所谓，列的次序可以任意交换
  • 任意两个元组的候选码不能相同
  • 行的顺序无所谓，行的次序可以任意交换
  • 分量必须取原子值

外码：

在这个表中不是码，但是在其他表中是主码．则成为是这个表的外码．

关系模型的三类完整性约束

实体性完整性和参照完整性

必须具备，成为关系的不变性

用户定义完整性

实体完整性

  • 关系的主属性不能去空值
  • 空值就是不知道或不存在或无意义

数据库的安全性

数据库的不安全因素

  • 非授权用户对数据库的恶意存取和破坏
  • 数据库中重要或敏感的数据被泄露
  • 安全环境的脆弱性

TCSEC/TDI标准的基本内容

从四个方面来描述安全性级别划分的指标
  • 安全策略
  • 责任
  • 保证
  • 文档
安全级别划分
  • A1 : 验证设计
  • B3: 安全域
  • B2: 结构化保护
  • B1: 标记安全保护
  • C2: 受控的存取保护
  • C1: 自主安全保护
  • D: 最小保护

数据库安全控制

常用方法

  • 用户身份鉴别

    • 系统提供的最外层安全保护措施

    • 用户标识：由用户名和用户标识号组成(用户标识号在系统整个生命周期内唯一)

      • 身份鉴别方法：
        • 静态口令鉴别：静态口令一般由用户自己设定，
        • 动态口令鉴别
        • 智能卡鉴别
        • 生物特征鉴别

  • 存取控制

    • 定义用户权限，并将用户权限登记在数据字典中

    • 合法权限检查

      • 用户发出存取数据库操作请求
      • DBMS查找数据字典，进行合法权限检查

    • 自主存取控制：

      • 用户对不同的数据对象有不同的存取权限
      • 不同的用户对同一对象也有不同的权限
      • 用户还可以将其拥有的存取权限转授给其他用户
      • SQL同伙GRANT与REVOKE语句实现

    • 授权与回收

      • GRANT: GRANT <权限>[,<权限>].. ON <对象类型><对象名> [,<对象类型><对象名>] TO<用户>[,<用户>]..[WITH GRANT OPTION];

      • [WITH GRANT OPTION]:指定该自居，可以再转授权限，没有指定，不能传播权限 。 不能循环授权

      • 发出GRANT:

        • 数据库管理员
        • 数据库对象创建者(即属主Owner)
        • 拥有该权限用户

      • 接受权限的用户

        • 一个或多个具体的用户
        • PUBLIC(及全体用户)

      • 例子：把查询Student权限授予用户U1

        • GRANT SELECT ON TABLE Student TO U1

  • 审计

    • 启用一个专用的审计日记，将用户对数据库的所有操作记录在上面

    • 审计员利用审计日志，监控数据库中的各种行为，发现非法存取，发现潜在威胁

    • 可以被审计时间：

      • 服务器事件

        • 审计数据库服务器发生的事件

      • 系统权限：

        • 对系统拥有的结构或模式对象进行操作的审计
        • 要求该操作的权限是通过系统权限获得的

      • 语句事件

        • 对SQL语句，如DDL,DML,DQL,DCL语句的审计

      • 审计日志管理：

        • 基本功能：提供了多种审计查阅方式

        • 多套审计分析和报表功能

        • 审计日志管理功能

          • 防止审计员误删记录，审计日志必须先转储后删除
          • 对转储的审计记录文件提供完整性和保密性保护
          • 只允许审计员查阅和转储审计记录，不允许任何用户新增和修改审计记录等

        • 提供查询审计设置即审计记录信息的专门视图

        • 审计功能的可选性

          • 审计很费时间与空间

        • 审计设置

          • AUDIT: 设置审计功能

          • NOAUDIT:取消审计功能

          • 用户级审计：任何用户可设置的审计，主要是用户针对自己创建的数据库表和视图进行审计

          • 系统审计：只能有数据库管理员设置，检测成功或失败的登录要求，检测授权和收回操作以及其他数据库级权限下的操作

          • 例子：

            • 对修改SC表结构或修改SC表数据的操作进行审计

              • AUDIT ALTER,UPDATE ON SC

  • 数据加密

    • 防止数据库中数据在存储和传输中失密的有效手段
    • 存储加密
      • 透明存储加密
        • 内核级加密保护方式，对用户完全透明
        • 将数据在写到磁盘是对数据进行加密，授权用户读取数据时在对其加密
        • 数据库的应用程序不需要做任何修改，只需要在创建语句中说明需加密的字段即可
    • 链路加密
      • 传输信息由报头和报文两部分组成
        • 报头：路由选择信息
        • 报文：要传送的数据信息
      • 报文和报头均加密：
    • 端到端加密
      • 在发送端加密，接收端解密
      • 只加密报文，不加密报头
      • 只需密码设备数量相对较少，容易被非法监听者发现并从中获取敏感信息
